### 4.1 架构 B：骑手支付 → 收入确认 → 商户留存

**业务目标**：骑手资金进入**持牌监管体系**下**运营商**二级商户；支付成功时 **1%** 分账至平台；周期完结后运营商留存净应分。

**参与方**：骑手、用户端、物联网、运营商收款商户、监管清分、平台商户、本后台。

**前置条件**：站点已绑定 `device_owner_id`；收款账户进件完成。

| 阶段 | 步骤 | 后台可见（运营商） | 是否打款 |
|------|------|-------------------------|----------|
| ① 收款 | 骑手购买包月/次卡，支付成功 | **我的流水 → 资金实收**（+预收） | 否 |
| ② 服务期 | 按日（包月）或按次/当笔（次卡）确认可分配收入 | **应分台账** 每日/每笔 +SplitLine | 否（仅记账） |
| ③ 换电 | 每次成功换电写入换电单，分摊应分 | **我的订单 → 换电**；应分「待结转」 | 否 |
| ④ 完结 | `valid_to` 届满 / 次卡用尽；无进行中退款 | 套餐状态 → **可打款** | 准备结算 |
| ⑤ 留存 | 周期完结；运营商留存净应分 | **打款批次** 已结算 | 是 |
| ⑥ 退款 | 中途退款：冲预收 + 负向 SplitLine | 实收退款出款；应分冲正 | 已打款则下周期抵扣 |

```mermaid
sequenceDiagram
  participant R as 骑手
  participant App as 用户端
  participant IoT as 物联网
  participant Mch as 收款二级商户
  participant Reg as 监管/清分
  participant BO as 运营后台
  participant Plat as 平台商户

  R->>App: 选站点/柜机购买套餐
  App->>Mch: 支付 P（架构 B）
  Mch->>Reg: 1% 分账至平台
  Mch->>BO: 资金实收 +P（预收）

  loop 服务期内（包月按日）
    BO->>BO: recognition_date 确认 P/D
    BO->>BO: 运营商净得 + 合作伙伴应分
    R->>App: 扫码换电
    App->>IoT: 开门/换电指令
    IoT-->>BO: 换电成功
    BO->>BO: 换电应分写入（待结转）
  end

  alt 周期完结 且 无未决退款
    BO->>BO: 套餐 payout=可打款
    BO->>BO: 汇总批次 Σ应分净额
    Reg-->>Mch: 商户留存净额
    BO->>BO: 打款批次=已分账
  else 中途退款 R
    App->>Mch: 退款至骑手
    Mch->>BO: 资金实收 -R
    BO->>BO: 停止未到期确认；冲正应分
    Note over BO: 已打款部分→下周期负向调整
  else 服务冻结
    Note over BO: 冻结期间不确认收入（建议）
  end
```

**异常分支**

| 场景 | 处理 |
|------|------|
| 支付成功但设备未入网 | 不计费；待物联网认证后人工补记或拒单 |
| 合作伙伴比例中途变更 | 仅**新确认日**起用新比例；历史 SplitLine 不追溯 |
| 有效站/SLA 未达标 | 批次可挂「冻结」，延迟打款 |
| 平台 SaaS 费 | 与运营应分**分表**；不向骑手套餐池计提 |

### 4.2 换电订单与应分（不即时打款）

**业务目标**：记录每次换电事实与过程日志，并把该次换电对应的经济份额记入应分，待套餐周期结束后一并结算。

**主流程**

1. 骑手在**服务中**且未冻结套餐下，于己方站点柜机扫码。
2. 用户端校验套餐有效性 → 物联网执行开门/换电 → 返回成功/失败。
3. 成功：生成 `SwapOrder`（关联套餐/额度池、`cabinet_id`、格口、电池 SOC；写入 **userOwner / cabinetOwner / batteryOwner** 三元组）。
4. 计算 **跨网设备服务费**：userOwner 向 cabinetOwner 付柜机费（C≠U）、向 batteryOwner 付电池费（B≠U）；相同则不产生该项；经平台代收/代付日清。
5. 按当日（或当次）已确认的套餐收入份额，计算运营商净得、合作伙伴应得（若站点已设合作方）→ 写入应分行，状态 **待结转**。
6. 套餐 `valid_to` 到达且无退款 → 该套餐下所有「待结转」→ **可打款** → 进入打款批次。

```mermaid
flowchart TD
  A[骑手扫码] --> B{套餐服务中且未冻结?}
  B -->|否| Z[换电失败/拒绝]
  B -->|是| C[用户端下发换电]
  C --> D[物联网执行柜机指令]
  D --> E{设备反馈成功?}
  E -->|否| Z2[失败+过程日志]
  E -->|是| F[换电订单成功]
  F --> G[过程日志: 用户操作/API/设备反馈]
  F --> T[三元组 U/C/B + 跨网设备服务费]
  T --> H[取当日已确认收入份额]
  H --> I[计算运营商净得/合作伙伴应分]
  I --> J[应分状态: 待结转]
  J --> K{套餐周期完结且无退款?}
  K -->|否| L[继续服务/换电]
  L --> A
  K -->|是| M[状态: 可打款]
  M --> N[纳入打款批次]
  N --> O[监管分账划出]

  subgraph BO [后台菜单]
```

**与套餐订单关系**

| 对象 | 字段/规则 |
|------|-----------|
| 套餐单 | `valid_from` / `valid_to`、`swap_used`、`accrued`、`payout` |
| 换电单 | 必须 `package_order_id`；`pay` 常为 0（包月内含） |
| 应分 | 不按换电笔即时打款，只累计至套餐周期批次 |

### 4.3 跨运营商清分（运营商往来账）

**业务目标**：个人用户或渠道成员跨站换电时，跨网设备服务费经平台代收/代付；**平时划扣保证金**，保证金为 0 才启用信用额度。

### 4.4 合作伙伴分润与提现

**业务目标**：运营商设站点、绑定设备后，可为站点添加**合伙人**参与**个人用户付费**分佣；合伙人可登录查看明细并提现。

**默认分佣**：平台 **1%** + 运营商 **99%**。站点绑定合伙人比例 **X%** 后：**平台 1% + 合伙人 X% + 运营商 (99−X)%**（例：X=30% → 1%+30%+69%）。**渠道人天用户不参与**。

**主流程（配置 → 计提 → 结算 → 提现）**

| 步骤 | 操作方 | 动作 | 后台位置 |
|------|--------|------|----------|
| 1 | 运营商主体 | 建站、绑定柜机+电池 | 站点 / 我的设备 |
| 2 | 运营商主体 | 员工 → 合作伙伴：名称、分佣比例（≤99%）、关联站点 | 员工 / 合作伙伴 Tab |
| 3 | 运营商主体 | 为站点指定唯一合伙人 | 合作伙伴分润 → **分润设置** |
| 4 | 系统 | 按**收入确认日**已生效规则（变更**次日 0:00** 生效）拆分三方份额 | 分润汇总 / 订单明细（**不脱敏**） |
| 5 | 系统 | 周期完结后结转可提现余额 | 合伙人 **个人账户** |
| 6 | 合伙人登录 | 绑卡、申请提现 | 个人账户 / 提现记录 |
| 7 | 运营商/财务 | 线下或代付打款后更新状态 | 提现：处理中 → 成功 |

```mermaid
flowchart TB
  subgraph cfg [配置阶段-运营商主体]
    S0[站点 + 柜机 + 电池]
    E1[员工-新增合伙人]
    E2[分佣比例 R% 占计提基数]
    E3[勾选关联站点]
    PS[分润设置: 每站绑定一位合伙人]
    S0 --> E1 --> E2 --> E3 --> PS
  end

  subgraph run [经营阶段-仅个人用户]
    O1[个人套餐/换电确认收入 B]
    O1 --> P1[平台 = round B×1% 3位]
    P1 --> P2[合伙人 = round B×R% 3位]
    P2 --> P3[运营商 = B − 平台 − 合伙人]
    P3 --> ST{payout 状态}
    ST -->|服务中| W1[待结转]
    ST -->|周期完结| W2[可打款/已结转]
  end

  PS --> run
  W2 --> ACC[合伙人个人账户余额+]

  subgraph cash [提现阶段-合伙人登录]
    C0{可提现 ≥ 申请额?}
    C0 -->|是| C1[提交提现]
    C1 --> C2[处理中]
    C2 --> C3[到账成功 扣余额]
    C0 -->|否| CX[拒绝/提示冻结]
  end

  ACC --> cash
  subgraph view [合伙人只读]
    V1[合作伙伴分润-明细/批次]
    V2[个人账户]
  end
  view --> ACC
```

**规则摘要**

- 一站点同一时间仅一位合伙人。
- 尾差：平台、合伙人各四舍五入至 **3 位小数**，余额归运营商。
- 分佣比例/站点绑定变更：**次日 0:00** 生效；待生效规则在「待生效规则」表展示。
- 汇总笔数须与明细可勾稽。
- 合伙人**不可**操作分润设置、员工管理、他人站点数据。

### 4.4b 换电范围策略

**业务目标**：运营商自主控制用户换电地理/主体范围，与平台信用额度停跨网叠加。

| 开关 | 关闭效果 |
|------|----------|
| **跨网换电** | ① 向本运营商付费的用户不可在其他运营商换电；② 其他运营商用户不可在本运营商站点换电 |
| **跨站点换电** | 同运营商下，用户仅可在**主绑定站点**换电 |

```mermaid
flowchart LR
  CFG[站点-换电范围设置] --> NET[cross_network_enabled]
  CFG --> SITE[cross_site_enabled]
  SCAN[骑手扫码] --> GATE{准入引擎}
  NET --> GATE
  SITE --> GATE
  CREDIT[信用额度停跨网] --> GATE
  GATE -->|通过| OK[换电流程]
  GATE -->|拒绝| TOAST[失败提示]
```

**后台位置**：运营商 → **站点** → 换电范围设置。  
细则见 [换电范围策略.md](./换电范围策略.md)。

### 4.5 员工 / 合作伙伴登录与权限

**业务目标**：同一后台支持经营主体、运营员工、合作伙伴三类身份；菜单与数据最小化暴露。

```mermaid

  Type -->|运营员工| Emp[读 employee.permissions]
```

### 4.6 套餐服务生命周期（后台可见）

**业务目标**：支持冻结、中途完结退款、到期完结；各状态驱动**是否继续确认收入**与**应分能否打款**。

| 状态 | 骑手侧 | 持有电池 | 收入确认 | 换电 | 应分/打款 |
|------|--------|----------|----------|------|-----------|
| 服务中 | 可换电 | 换电后持有；冻结申请前须还电 | 按日确认 | 正常累计待结转 | 周期未完结为待结转 |
| 已冻结 | 不可换电 | **不持有**（冻结前须还电） | **建议暂停**确认 | 拒绝新换电 | 待结转维持 |
| 中途完结 | 服务终止 | **不持有**（退款前提已还电） | 停止未到期确认；冲已确认 | 停止 | 冲正后重新结算或关闭 |
| 待首换开通 | 未首换 | **不持有** | — | 首次换电后开通 | — |
| 已完结 | 到期 | 须已还电 | 确认至 valid_to | — | 待结转→可打款 |

```mermaid
stateDiagram-v2
  [*] --> 服务中: 支付成功<br/>预收+P

  服务中 --> 已冻结: 骑手申请冻结<br/>须未持有电池
  已冻结 --> 服务中: 解冻通过<br/>有效期顺延

  服务中 --> 中途完结: 主动结束
  中途完结 --> 退款处理: 退未使用套餐费
  退款处理 --> 押金处理: 非免押须还电退押
  押金处理 --> [*]

  服务中 --> 已完结: valid_to 到达
  已完结 --> 可打款: 无未决退款
  可打款 --> 已打款: 分账完成
  已打款 --> [*]

  中途完结 --> 冲正应分: 负向 SplitLine
  冲正应分 --> [*]

  note right of 已冻结
    后台: 我的订单-服务变更
    套餐详情展示 freezeInfo
  end note
```

**中途完结退款拆分（后台「服务变更」）**

1. 骑手发起 → 运营商审核（演示可简化）。
2. 计算应退套餐费 R（剩余天数法等）。
3. 资金实收退款出款；应分台账冲正。
4. 若有电池押金且非信用免押：还电验收后押金原路退。

### 4.7 租赁设备与月租金

**业务目标**：资金方出租柜/电；承租方（运营商）按月付租；可用站点经营收入自动扣划，不足则人工跟进。

**主流程**

1. **资金方**：维护 `LeaseContract`（承租方、设备清单、月租金、押金、租期）。
2. **承租方**：「协议与设备」查看绑定资产；「月租金」见各期账单与站点收入覆盖。
3. **每月**：生成 `LeaseRentBill`（应还、站点收入、缺口、缴纳方式）。
4. **自动扣款**：若开通且 `站点可分配收入 ≥ 月租金` → 从租金支付账户扣至出租方。
5. **收益不足**：自动扣款失败 → 状态需**人工处理**（补缴/登记跟进）。
6. **资金方**：「租金收缴」看实还进度。

```mermaid
flowchart TD
  L[资金方] --> C[签订租赁协议]
  C --> BIND[设备清单绑定承租方]
  BIND --> BILL[生成月租金账单]

  BILL --> CHK{当月站点收入 ≥ 应付租金?}
  CHK -->|是且已开通自动扣| AUTO[自动扣款至出租方账户]
  CHK -->|否或未开通| GAP[收益不足/待人工]
  GAP --> MAN[承租方手动缴纳或跟进登记]
  AUTO --> REC[出租方-租金收缴-已还]
  MAN --> REC

  subgraph lessee [承租方后台]
    V1[协议与设备]
    V2[月租金-覆盖与缺口]
  end
  BIND --> V1
  BILL --> V2
```

**与订单关系**：租赁设备仍记 `device_owner_id` = 承租运营主体；骑手订单归属不变，租金从经营收入侧扣划。

### 4.9 三台账与打款批次（对账主线）

**切勿混列**：骑手「付了多少钱」≠「已分给各方多少钱」≠「银行已划出多少钱」。

| 台账 | 含义 | 典型条目 | 菜单 |
|------|------|----------|------|
| 资金实收 | 监管户实收实付 | 套餐支付 +、退款 −、押金收退 | 我的流水 → 资金实收 |
| 应分台账 | 确认收入与各方应得/冲正 | SplitLine +/−、待结转/可打款 | 我的流水 → 应分台账 |
| 打款批次 | 实际结算记录 | 批次号、商户留存净额 | 我的流水 → 打款批次 |

```mermaid
flowchart LR
  PAY[骑手支付] --> CASH[资金实收 +预收]
  CASH --> REC[按规则确认收入]
  REC --> ACC[应分台账 +各方应分]
  SWAP[换电成功] --> ACC
  ACC --> WAIT[待结转]
  WAIT --> OK[可打款]
  OK --> BATCH[打款批次]
  BATCH --> CASH2[监管分账划出]
  REF[退款] --> CASH3[资金实收 -退款]
  REF --> ACC2[应分冲正 -]
```

**打款批次生成条件（套餐维度）**

1. `valid_to` 已过或次卡额度用尽，订单关闭。  
2. 无进行中 `refund`。  
3. 可选：运营商在后台确认，或 T+N 自动确认。  
4. Σ应分净额 > 0 且未冻结。  

**合作伙伴分润** 的「结算批次」与运营商「打款批次」概念对应，视角为合作方应付。

### 4.10 企业天数池

**业务目标**：渠道向一家或多家签约运营商批量买人天（**一运营商一池**）→ 团队登记骑手 → 换电消耗（**可跨网**）→ 离职回池。细则见 **[天数池.md](./天数池.md)** §10。

**首版配置（拍板 2026-06-12）**

| 配置 | 首版 | 运行要点 |
|------|------|----------|
| `deduct_mode` | 自然日 | 冻结日不计 |
| `billing_mode` | 预占确认 | 00:00 预占 → 换电或持电池确认 → 否则日终释放 |
| `activation_mode` | 分配即开通 | 首换开通第二期 |
| `pool_expiry_refund_policy` | 允许部分过期可退 | B2B 退款协商后运营商扣减 |
| B2B 资金 | 采购时结清 | 渠道采购款到账即运营商收入；**消耗不向运营商二次打款** |
| 平台 1% | 确认消耗计提 | 标准人天价 × 1% 代扣额度售卖方 U |

**硬性规则**：骑手**离职** → 终止池来源 `PackageOrder` → **剩余天数强制回池**（不退骑手）。

```mermaid
flowchart TD
  A[渠道采购付款] --> B[运营商已收款 + DayPool 可用天数]
  B --> C[分配给骑手]
  C --> D[PackageOrder 服务中]
  D --> E[00:00 预占 1 人天]
  E --> F{换电或持电池?}
  F -->|是| G[确认消耗 -1 人天]
  F -->|否| H[日终释放预占]
  G --> I[平台向 U 计提 1%]
  D --> J[离职]
  J --> K[剩余天数回池]
```

与 §4.2 换电、§4.6 冻结/中途完结共用。**打款批次（§4.9）仅适用于 C 端个人套餐**，不适用于渠道人天池消耗。B 端退款对象为企业而非骑手。
